Array vs. Linked List

Comprehensive Study Notes & Visualizations from the Lecture

🧐 Introduction: Which is Better?

The instructor emphasizes a fundamental rule of data structures right at the start: We cannot universally say that an Array is better or a Linked List is better.

The "best" choice depends entirely on your specific requirements:

  • The size of the data you are dealing with.
  • Which operations you need to perform most frequently (e.g., Are you mostly reading data? Or are you constantly inserting/deleting data?).

To decide, we must compare them across several different core factors.

⏱️ Factor 1: Cost of Accessing an Element

🟦 Arrays

Arrays allow for Random Access (or Direct Access). Because elements are stored in contiguous (consecutive) memory locations, the compiler can instantly calculate the exact address of any element.

💾 Factor 2: Memory Requirement vs. Utilization

Note: The instructor emphasizes that Requirement and Utilization are two different concepts.

🟦 Arrays: Fixed Size

You must declare the maximum limit of the array at compile time. This leads to poor memory utilization if not filled.

  • Utilization (Poor): In the visual above, size is 7, but only 3 elements are stored. 7 slots * 4 bytes = 28 bytes allocated. 16 bytes are completely wasted.
  • Requirement (Low): Despite wasted space, arrays only store data. No extra space is needed for pointers.

🟩 Linked Lists: Dynamic Size

Nodes are created at runtime exactly when needed. This leads to excellent memory utilization (no empty slots).

  • Utilization (Excellent): If you need 3 elements, you create exactly 3 nodes. 0 wasted space.
  • Requirement (High): Every node requires space for Data AND a Pointer.
    Example: Integer (4 bytes) + Pointer (4 bytes) = 8 bytes per node.
    For 3 elements: 3 * 8 = 24 bytes.
💡 The "Complex Data" Scenario:
What if you are storing large objects (e.g., 16 bytes) instead of simple integers?
- Array (size 7): 16 bytes * 7 slots = 112 bytes.
- Linked List (3 elements): 3 nodes * (16b data + 4b pointer) = 60 bytes.
Conclusion: For large data types where you might not fill the array, Linked Lists save massive amounts of memory!

⚡ Factor 3: Cost of Insertion & Deletion

The time complexities for insertion and deletion are identical. The instructor breaks this down by location:

Location Array (🟦) Linked List (🟩)
At the Beginning O(n)
Must manually shift EVERY existing element one space to the right to make room. 		O(1)
Simply create a node, point it to the current head, and update the head pointer.
At the End O(1)
If not full, you instantly know the next empty index and drop data directly in. 		O(n)
Must traverse the entire list sequentially to find the last node (pointing to Null).
At i-th Position (Middle) O(n)
Must shift all elements to the right of the target. On average, n/2 shifts. 		O(n)
Must traverse from the head to reach the i-th position. On average, n/2 traversals.

🛠️ Factor 4 & 5: Ease of Use & Searching

Factor 4: Ease of Use

Arrays are generally easier to use and implement. You don't have to worry about manual memory allocation, managing pointers, or accidentally causing memory leaks by breaking a chain of addresses.

Linked Lists are harder to implement because they require careful pointer manipulation and dynamic memory management.

Factor 5: Searching

Arrays support Linear Search. Furthermore, if the array is sorted, it supports Binary Search because the math formula allows you to instantly access the exact middle element.

Linked Lists ONLY support Linear Search. Even if sorted, you cannot perform Binary Search because you cannot directly jump to the middle node; you still must traverse from the head.

📋 Summary Cheat Sheet

Feature Array Linked List
Size Fixed (Compile-time) Dynamic (Run-time)
Memory Allocation Contiguous (consecutive blocks) Random (linked via pointers)
Accessing Elements O(1) (Random access) O(n) (Sequential access only)
Memory Utilization Inefficient (Prone to wasted space) Highly Efficient (No wasted space)
Memory Requirement Less (Data only) More (Data + Pointers)
Insert/Delete at Beginning O(n) O(1)
Insert/Delete at End O(1) O(n)
Insert/Delete in Middle O(n) O(n)
Searching Algorithms Linear & Binary Search Linear Search only
