2. The Baseline: Standard Fixed-Length Coding

Before compressing, we must understand how much space the message normally takes. Computers normally store text using ASCII, where every character is converted to an 8-bit binary number.

  • A = ASCII 65 = 01000001
  • B = ASCII 66 = 01000010
Total Size = 20 characters × 8 bits
Baseline Cost = 160 bits

3. Attempt 1: Custom Fixed-Length Coding

Since we only have 5 unique characters, we don't need 8 bits per character. 3 bits gives us 8 combinations (23 = 8), which is enough for our 5 letters.

Custom Codes: A=000, B=001, C=010, D=011, E=100

The Catch (The Decoding Table):

If you just send the compressed message, the receiver has no idea what 000 means. You must also send the decoding table along with the message so they can read it.

Message = 20 chars × 3 bits = 60 bits
Table = (5 chars × 8 bits) + (5 codes × 3 bits) = 55 bits

Total Custom Cost = 60 + 55 = 115 bits

Better than 160, but we can still improve!

4. The Solution: Huffman Coding

Core Concept (Variable-Length): Give shorter codes to characters that appear often, and longer codes to characters that rarely appear.

5. The Final Huffman Tree

Starting from the Root (20): Write a 0 on every Left branch, and a 1 on every Right branch. Note that all characters end up as "Leaf Nodes" at the bottom.

Tracing the Codes (Top to Bottom):

  • E: Left ➔ Left = 00
  • A: Left ➔ Right = 01
  • B: Right ➔ Right = 11
  • D: Right ➔ Left ➔ Left = 100
  • C: Right ➔ Left ➔ Right = 101

6. Huffman Compression Final Math

Notice that B, which appears the most (7 times), got a short 2-bit code. D, which appears the least (2 times), got a longer 3-bit code.

Message Size (Freq × Code Length):
A: 4 × 2 = 8 bits
B: 7 × 2 = 14 bits
C: 3 × 3 = 9 bits
D: 2 × 3 = 6 bits
E: 4 × 2 = 8 bits
Message Total = 45 bits

Table Size (Characters + Sent Codes):
Chars: (5 × 8 bits) = 40 bits
Codes: (2+2+3+3+2) = 12 bits
Table Total = 52 bits

Grand Total (45 + 52) = 97 bits 🏆

7. The Prefix Rule (Crucial Concept)

Why doesn't a variable-length code get confusing when the receiver tries to read it?

"No code is a prefix of another code."

Example of a BAD Code: If A = 0, B = 1, and C = 01. If the receiver gets 001, does it mean A-A-C or A-B? It's ambiguous.

Why Huffman is GOOD: Look at our codes: 00, 01, 11, 100, 101. None of these codes forms the exact beginning of any other code. Because characters only exist at the Leaf Nodes, you never pass *through* a character to get to another character!

8. Instructor Summary Notes

  • History: Developed by David Huffman in 1951.
  • Rule: Encoding strictly follows the Prefix Rule, ensuring no ambiguity during decoding.
  • Core Logic: The most generated character gets the smallest code; the least generated gets the largest code.
  • Time Complexity: O(N log N), where N is the number of unique characters being sorted and placed into the tree.
