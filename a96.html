Binary Search Algorithm

Complete Whiteboard Breakdown & Study Notes

1. The Core Concept

Binary Search is a Divide and Conquer algorithm. Instead of checking elements one by one from start to finish (Linear Search), it continuously cuts the search space in half. Because of this, it takes significantly less time to find data in large arrays.

🚨 THE GOLDEN RULE (Prerequisite):
The array MUST BE SORTED. If the data is not sorted, you cannot apply Binary Search. You must sort the array first.

2. Variables & The 3 Rules

To navigate the array, we use three pointers (variables):

  • l (left): Points to the start index of our current search space.
  • r (right): Points to the end index of our current search space.
  • mid: The middle index. Calculated as: Floor of (l + r) / 2

The 3 Checking Cases

Once we calculate the `mid` index, we compare our target data to a[mid]. Only one of three things can happen:

Case 1: data == a[mid]
Target found! Stop searching and return the mid index.
Case 2: data < a[mid]
Target is smaller. It must be on the left side. We ignore the right side by moving
r = mid - 1.
Case 3: data > a[mid]
Target is larger. It must be on the right side. We ignore the left side by moving
l = mid + 1.

4. Walkthrough 2: Unsuccessful Search

What happens if the data is not in the array? When does the algorithm know to stop?

Goal: Search for data = 60

Let's fast forward through the steps we already know:

  • Step 1: mid = 4 (val 25). 60 > 25. Move l = 5.
  • Step 2: mid = 7 (val 63). 60 < 63. Move r = 6.
  • Step 3: mid = 5 (val 45). 60 > 45. Move l = 6.
  • Step 4: mid = 6 (val 59). 60 > 59. Move l = mid + 1 ➡️ l = 7.
🚨 The Stopping Condition:
At this point, l = 7 and r = 6. Notice that L is now greater than R. The left and right pointers have crossed each other. This is the official stopping condition. If l > r, it mathematically proves the data is not present in the array. Return -1.

5. The Algorithm Code

Here is the C-style pseudocode written on the whiteboard, translating the logic above into instructions:

int BinarySearch(int a[], int n, int data) {
    int l = 0;              // Start pointer
    int r = n - 1;          // End pointer
    
    // Keep looping as long as left hasn't crossed right
    while (l <= r) {        
        int mid = (l + r) / 2;    // Find the middle (takes integer floor automatically in C)
        
        if (data == a[mid]) {
            return mid;           // Case 1: Found it! Return the index.
        }
        else if (data < a[mid]) {
            r = mid - 1;          // Case 2: Data is smaller, search left half.
        }
        else {
            l = mid + 1;          // Case 3: Data is larger, search right half.
        }
    }
    
    // If the while loop finishes and breaks (l > r), the data wasn't found
    return -1; 
}

6. Time Complexity Analysis

Why is Binary Search so much better than Linear Search? Let's analyze the math behind the while loop.

Best Case: O(1)

This happens if the element you are looking for happens to be at the exact middle of the array on your very first guess. The loop runs exactly once.

Worst/Average Case: O(log n)

Think about how the array size shrinks. In the worst case, you keep dividing the array by 2 until only 1 element is left.

  • If `n = 100`, the search space sizes go: 100 ➡️ 50 ➡️ 25 ➡️ 12 ➡️ 6 ➡️ 3 ➡️ 1.
  • It only takes a maximum of 7 steps to search an array of 100 elements. (A linear search could take 100 steps!)

The Math Derivation (From the Board)

Let k be the number of steps (divisions by 2) it takes to reach 1 element.

n / 2k = 1

n = 2k

log2(n) = log2(2k)

k = log2(n)

Therefore, the algorithm takes exactly log2(n) steps. For an array of 100 elements, log2(100) is approximately 6.64. We round up to 7 steps maximum. This logarithmic behavior makes binary search incredibly powerful for massive datasets.
