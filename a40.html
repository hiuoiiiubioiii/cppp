Binary Expression Trees

Constructing from Postfix Expressions

Scroll to learn ↓

1 Why Postfix? (The Context)

In programming, constructing an expression tree from an Infix expression (e.g., a + b * c) is highly complex. The computer has to account for mathematical rules like BODMAS/PEMDAS (Operator Precedence) and Associativity.

The Solution: Convert the Infix expression to a Postfix expression first. Postfix natively ignores precedence rules and parentheses, allowing us to build the tree purely by reading left-to-right using a Stack data structure.

Rule of Thumb for Binary Expression Trees:
• Leaf Nodes (bottom) are always Operands (variables like a, b, c).
• Internal Nodes (branches) are always Operators (+, -, *, /).

2 The Algorithm & Golden Rules

We read the Postfix string strictly from Left to Right. We use a Stack to temporarily hold pointers to the tree nodes we create.

Rule 1: If it's an OPERAND (a, b, c...)

  • Create a new tree node containing that operand.
  • Push the pointer of this new node into the stack.

Rule 2: If it's an OPERATOR (+, -, *, /)

  • Pop two pointers from the stack.
  • ⚠️ CRITICAL: Top popped item = RIGHT child. Next popped item = LEFT child.
  • Create a node for the operator. Attach the left and right children.
  • Push the pointer of this new Subtree back into the stack.

The Whiteboard Example

The instructor writes the following expressions. We will only use the Postfix version.

Infix: a * b / c + e / f * g + k - x * y
Postfix: a b * c / e f / g * + k + x y * -

Phase 1: Processing a b * c /

1. Scan a: Operand. Create node, push to stack.

2. Scan b: Operand. Create node, push to stack.

3. Scan *: Operator. Pop b (Right), Pop a (Left). Create *, attach, push subtree.

4. Scan c: Operand. Create node, push.

5. Scan /: Operator. Pop c (Right), Pop * subtree (Left). Attach, push to stack.
