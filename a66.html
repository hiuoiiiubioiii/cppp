Red-Black Tree Deletion

1. Prerequisites & The Core Problem

Before deleting, we must remember the 3 Golden Rules of a Red-Black Tree that keep its operations at O(log n):

  • Root Property: The root node must always be Black.
  • No Red-Red Conflict: If a parent is Red, its children MUST be Black.
  • Black Height Property: Every path from a node down to its descendant NIL nodes must contain the exact same number of Black nodes.

⚠️ Why is Deletion Tricky?

Deleting a Red node is easy—it doesn't affect the Black Height. But deleting a Black node breaks Rule #3 by reducing the black-height of that specific path by 1. We have to fix this imbalance.

2. Step One: Standard BST Deletion

Before applying any Red-Black rules, you must delete the node using standard Binary Search Tree (BST) logic.

  • Target is a Leaf: Simply delete it.
  • Target has One Child: Replace the target node with its child, then delete the target.
  • Target has Two Children: Find the In-order Successor (smallest value in the right subtree). Copy its value to the target node, then delete the original successor node.

*Note: The instructor always uses the In-Order Successor for two-child deletions in this lecture.*

3. The Simple Deletion Cases

Simple Case A: Delete a RED Leaf

If the node to be deleted is a Red Leaf, just delete it. No properties are violated.

Simple Case B: Black Node with ONE Red Child

If you delete a Black node that has one Red child, the Red child moves up to replace it. Rule: The child must take the color of the node it replaced. So, color it Black.

4. The Problem: "Double Black" (DB)

What happens if you delete a Black Leaf node?

It is replaced by a NIL node. To prevent the Black Height from changing, the deleted node passes its "blackness" down to the `NIL` node. Since `NIL` is already Black, it absorbs the extra blackness and becomes a Double Black (DB) node.

A Double Black node is illegal. The goal of the algorithm is to convert the DB node back into a Single Black node.

5. Algorithm Terminology

To resolve a DB, you must ALWAYS look at its Sibling.

DB: The Double Black node.
Sibling (S): The node sharing the same parent as DB.
Near Child: The child of the sibling closest to the DB node. (If DB is left, Near child is Sibling's left child).
Far Child: The child of the sibling furthest from the DB node.

Case 1 & 2: The Easy Outs

Case 1: Node to delete is RED.

Just delete it. We already covered this.

Case 2: Root is Double Black.

If the DB node bubbles all the way up to become the Root, simply remove the Double Black and make it a Single Black root. The black height of the entire tree uniformly decreases by 1.

Case 3: Sibling is BLACK, both Sibling's children are BLACK

Action: Remove DB status (make it single black). Make the Sibling Red. Add a layer of "Blackness" to the Parent. If parent was Red, it becomes Black (done). If parent was Black, it becomes Double Black (re-apply cases to parent).

Case 4: Sibling is RED

Action: Swap colors of Parent and Sibling. Rotate Parent in the direction of the DB. Result: DB is not resolved, but it shifts the tree into Case 3, 5, or 6. Re-apply rules.

Case 5: Sibling is BLACK, "Near" child RED, "Far" child BLACK

Action: Swap colors of Sibling and Near Child. Rotate Sibling away from the DB. Result: This perfectly transforms the tree into Case 6. Proceed immediately to Case 6.

TERMINAL CASE

Case 6: Sibling is BLACK, "Far" child is RED

Action: Swap colors of Parent and Sibling. Color the Far Child Black. Rotate Parent in the direction of the DB. Remove DB status. Problem completely solved!

6. Master Walkthrough

Follow the mental loop: 1. Standard Delete -> 2. Was it red? (Done) -> 3. Was it black? (Place DB) -> 4. What color is Sibling? -> 5. Apply Case.

Operation 1: Delete 55 (Red Leaf)

It is a Red leaf node. Apply Case 1. Just delete it. The tree remains balanced.

Operation 2: Delete 15 (Black Leaf)

  • Deleting 15 leaves a `NIL` Double Black node.
  • Check Sibling: Sibling is `25`, which is Red.
  • Match: This is Case 4.
  • Action: Swap colors of Parent (20) and Sibling (25). Rotate Parent left.
  • Re-check: DB still exists. New sibling is `28(B)`. Both of `28`'s children are `NIL` (Black).
  • Match: This is now Case 3.
  • Action: Make Sibling (28) Red. Remove DB. Pass blackness to Parent (20). Since Parent is Red, it absorbs it and becomes Black. Problem Solved.

Operation 3: Delete 80 (Black Leaf)

  • Deleting 80 leaves a `NIL` Double Black node.
  • Check Sibling: Sibling is `65(B)`. Sibling's "Near" child is `70(R)`. Sibling's "Far" child is `NIL(B)`.
  • Match: Near is Red, Far is Black. This is Case 5.
  • Action: Swap colors of Sibling (65) and Near child (70). Rotate Sibling right.
  • Re-check: Sibling is now `70(B)`, and "Far" child is `65(R)`.
  • Match: Far is Red. This perfectly transforms into Case 6.
  • Action: Swap colors of Parent (68) and Sibling (70). Color Far child (65) Black. Rotate Parent left. Remove DB. Problem Solved.
