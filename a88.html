Detecting a Cycle in an Undirected Graph (BFS)

In this lecture, the instructor explains how to determine if a cycle exists in an undirected graph. While there are multiple methods to achieve this (DFS, Topological Sort, Disjoint Sets), this specific lesson breaks down the Breadth-First Search (BFS) approach.

Key Data Structure: Because we are using BFS, which explores the graph level-by-level, we must use a Queue (First-In-First-Out).

The Flag System (State Management)

To keep track of where we are in our BFS traversal, we assign a "Flag" (or state) to every node. By observing changes in these flags, we can catch when a cycle occurs.

-1 : Unvisited (Not yet discovered)
0 : In Queue (Discovered, waiting)
1 : Visited (Fully processed)

Step 5: Processing Node D (Cycle Detected!)

  • Dequeue D: Move to Visited Set. Flag becomes 1.
  • Check Neighbors: D connects to B and E.
  • Ignore B (Flag is already 1).
  • Look at E: Node E currently has a flag of 0.
The "Aha!" Moment: What does a flag of 0 mean? It means E is already in the queue! It was put there previously by Node B. Because we are reaching Node E again from a completely different path (via Node D), a cycle exists!

The Core Logic & Pseudocode

To implement this in code, you simply modify your standard BFS algorithm with two conditional checks when evaluating the neighbors (let's call a neighbor v) of the current node.

The Golden Rule: "If any vertex finds its adjacent vertex with flag 0, then it contains a cycle." 
// Pseudocode snippet for checking neighbors (v)

if (f[v] == 0) {
    // If neighbor is already in the queue, we found a second path.
    return 1; // Cycle Exists!
} 

if (f[v] == -1) {
    // If neighbor is undiscovered, add to queue.
    f[v] = 0; // Update flag to 'In Queue'
    insert(v); // Push to Queue
}
