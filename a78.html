🌳 Foundation: The Rules

Before we delete anything, the instructor writes the foundational math on the top right of the whiteboard. You must memorize these constraints for the tree to remain valid.

Order (m) = 4
Max Children = 4
Min Children = ⌈m/2rceil = ⌈4/2rceil = 2
Max Keys (Data) = m - 1 = 3
Min Keys = ⌈m/2rceil - 1 = 1

Crucial Rule: Data in a B+ Tree is present in the leaf nodes only. Internal nodes only contain indexes to guide the search. Leaf nodes are connected horizontally like a linked list.

The Search: Root `[20]` → Go Right → `[25 | 31]` → Go Left → Found in leaf `[20 | 21]`.

"This is the simplest case. The node having the data you want to delete has more than the minimum number of keys. You simply delete that key. No need to do anything else."

The node has 2 keys. Our rule says Min Keys = 1. Therefore, 2 - 1 = 1. We are perfectly safe.

Case 3 Delete 20: Leaf Underflow (Borrow Right)

The Search: Root `[20]` → Equal to index → Found in leaf `[20]`.

"If you delete 20, the node is having 0 keys. That is not possible... It will ask its right sibling, 'Can you give me one key?' This will say yes, because it is having keys more than the minimum."

Step 1: Delete 20. Node becomes empty `[ ]`.
Step 2: Borrow the minimum key from the right sibling `[25 | 28]`. We borrow `25`.
Step 3: Because 25 shifted to the left node, the right node now starts at 28. We must update the parent index to reflect this new boundary.

Case 5 Delete 7: Leaf Underflow (Borrow Left)

The Search: Found in leaf `[7]`.

"It will ask its left sibling... 'Yes', because it has more than minimum keys. It will borrow the MAXIMUM value from the left sibling."

Step 1: Node underflows `[ ]`. We look left to sibling `[1 | 4]`.
Step 2: When borrowing from the left, we take the maximum value. We borrow `4`.
Step 3: The parent index separating these was `7`. Since 4 moved to the right node, the new right node is `[4]`. The parent index updates to `[4 | 17]`.

Case 6 Delete 25: Merging Leaves

The Search: Found in leaf `[25]` (which is also the root index).

"It will ask its sibling, 'Can you give me 1 key?' It will say NO, because it is also having minimum number of keys. Now we cannot borrow. We will merge these nodes."

Step 1: Delete 25. Node is empty `[ ]`. Right sibling `[28]` has only 1 key. Cannot borrow.
Step 2: Merge! Combine `[ ]` and `[28]` into a single node `[28]`.
Step 3: Because two nodes became one, the index separating them in the parent `[28 | 42]` must be destroyed. Delete 28. Parent becomes `[42]`.
Step 4: Update the root index `[25]` to `[28]`.

Case 7 Delete 42: Internal Node Underflow

The Underflow Chain Reaction:

  1. Delete 42. Leaf becomes empty. Left sibling `[28]` has 1 key. Merge them into `[28]`.
  2. Delete separating index `42` from parent. Parent internal node becomes empty `[ ]`!

"This node is having only one child, but minimum children should be 2. Min keys is 1... Now what will you do? It will borrow from its left sibling internal node."

Internal Borrowing Rule: We pull the parent (root) down, and push the sibling's max key up.

The Final Chain Reaction:

"It will ask its right sibling internal node. That is also having minimum number of keys. In that case, what will happen? These two should be merged with their parent... So this is now shrinking of the height."
