Part 1 Rules & Specifications

Before deleting anything, you must know the specific mathematical rules governing the B-Tree you are working with. The instructor writes these limits based on the Order of the tree.

Tree Order (m) = 5

  • Max Children = m = 5
  • Min Children = ⌈m/2⌉ = ⌈2.5⌉ = 3
  • Max Keys per node = m - 1 = 4
  • Min Keys per node = ⌈m/2⌉ - 1 = 2

Crucial Rule: Every node in this tree (except the overall root) MUST have a minimum of 2 keys. If deleting a key drops a node to 1 key, it creates an "underflow" violation, and you must Borrow or Merge to fix it.

Case 1 Target Key is in a Leaf Node

When the key you want to delete is found in a leaf node (a node at the very bottom with no children), you evaluate the situation based on how many keys that leaf currently holds.

Subcase 1A: Simple Deletion

Scenario: The target leaf node contains 3 or 4 keys. Because the minimum is 2, you can safely remove the key without violating any rules.

Subcase 1B: Borrowing from Immediate Left Sibling

Scenario: The target leaf has exactly 2 keys (the minimum). Deleting one leaves only 1 key (invalid). However, its immediate left sibling has >2 keys, so it can lend one.

Action: Delete 23. Target is [23, 27]. Left sibling is [14, 15, 16].

The Mechanism: Borrowing ALWAYS goes through the parent. Take the maximum value from the left sibling (16), move it UP to the parent. Take the separating value from the parent (20), move it DOWN to the target. Finally, delete 23.

Subcase 1E: The "Cascading Problem" during Merging

Scenario: You perform a merge, pulling a key down from the parent. But this causes the parent internal node to fall below the minimum 2 keys!

Action: Delete 20. This causes parent node [10, 16] to drop its '16' down, leaving only [10]. An internal node with 1 key is a violation.

The Fix: Apply the exact same rules to the broken internal node. It tries to borrow from its sibling [70, 77]. Sibling only has 2 keys. Therefore, the internal nodes must MERGE with the Root [50].

Case 2 Target Key is in an Internal Node

If the key you want to delete is NOT a leaf, you cannot just delete it and leave an empty hole in the middle of the tree. You must replace it with a value from further down the tree.

Replacement Rules:

  • In-order Predecessor: The largest value in the left subtree. (Go left once, then all the way right).
  • In-order Successor: The smallest value in the right subtree. (Go right once, then all the way left).
  • Condition: You can only "steal" a predecessor or successor if its leaf node currently has >2 keys.

Using Predecessors / Successors

Action 1: Delete 70. Left child is [51, 52, 60, 68]. Because it has 4 keys (>2), we steal the predecessor (68) to replace 70.

Action 2: Delete 95. Left child has 2 keys (can't use). Right child is [100, 110, 111]. It has 3 keys (>2). We steal the successor (100) to replace 95.

Part 4 Shrinking the Height of the Tree

If you perform a long sequence of deletions that result in cascading merges all the way up to the Root node, the tree can physically shrink in height.

Scenario: After extensive deletions, the root is [80]. Its left child is [10]. Its right child is [90]. The children only have 1 key (a violation), and they cannot borrow from each other.

The Fix: They must merge with the Root node itself.

Because the old root (80) was pulled down into the merge, it disappears as an independent level. The newly merged node [10, 80, 90] becomes the entirely new root of the tree, shrinking the overall height.
